Infrastructure of JenkinsThe infrastructure of Jenkins consists of two main components: the Jenkins master server and Jenkins agents (also known as slaves). This distributed architecture allows Jenkins to efficiently handle multiple builds and perform tasks in parallel.
7
Jenkins Master Server The Jenkins master server is the core component of the Jenkins infrastructure. It is responsible for managing the entire CI/CD process, coordinating build jobs, and handling the web-based user interface. Some key functions of the Jenkins master server include:
Job Execution: The master server schedules and assigns build jobs to Jenkins agents for execution based on predefined pipelines or triggers.

User Interface: Jenkins provides a web-based user interface accessible through a web browser, allowing users to configure, manage, and monitor jobs and build pipelines.

Job Configuration: Users can define and configure build jobs, pipelines, and other automation tasks using the Jenkins web interface or by writing Jenkinsfiles (declarative pipeline scripts) to define pipeline stages.

Plugin Management: The master server manages the installation, updates, and configuration of plugins, which enhance Jenkins with additional functionality and integrations.

Distributed Builds: It coordinates the distribution of build tasks across Jenkins agents for parallel execution, optimizing resource utilization and build efficiency.
8
Jenkins Agents (Slaves)Jenkins agents are worker nodes that perform the actual build and deployment tasks assigned by the master server. Agents can be either physical machines or virtual instances that are connected to the Jenkins master server over a network. Key characteristics of Jenkins agents include:
Build Execution: Agents execute build tasks, tests, and deployments as directed by the Jenkins master.

Platform Diversity: Jenkins agents can run on different operating systems, allowing teams to build and test their projects on various environments.

Scalability: The number of agents can be dynamically scaled up or down based on the workload and the need for parallel execution.

Node Labels: Agents can be labeled with specific tags, enabling the Jenkins master to select agents with matching labels to execute particular jobs.

Security: Agents communicate securely with the Jenkins master, and administrators can configure access controls to limit the actions agents can perform.

Offline/Online Status: Agents can be taken offline or brought back online based on maintenance requirements or resource availability.

Types of Jenkins In Jenkins, agents (also known as slaves) are the worker nodes responsible for executing build, test, and deployment tasks as directed by the Jenkins master server. There are different types of agents based on how they are configured and where they run:

Permanent Agents:
Permanent agents are long-lived worker nodes that are always online and connected to the Jenkins master.
These agents are pre-configured and available to handle build tasks at any time.
Permanent agents are suitable for stable environments where there is a consistent need for build and deployment capacity. These are suitable when there is a steady demand for build and deployment capacity. They are ideal for organizations with a stable infrastructure where agents can be consistently available.
eg : LINUX, WINDOWS.

Cloud Agents:
Cloud agents, also known as dynamic agents, are created on-demand in cloud environments (e.g., Amazon EC2, Microsoft Azure, Google Cloud Platform) by Jenkins to handle fluctuating build workloads.
These agents are automatically provisioned when needed and terminated when the build or deployment tasks are completed.
Cloud agents allow Jenkins to scale up or down the build capacity based on the workload, saving resources when agents are not in use. Cloud agents are beneficial when the workload varies over time or when there is a need for temporary build capacity. They are cost-effective since resources are provisioned only when needed and terminated when not in use.
eg : DOCKER, AWS, KUBERNETS.
